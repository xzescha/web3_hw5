# Gas optimization quizz

Homework for students on the topic of gas optimization.

Any tricks can be used in the solution, the main thing is to reduce the cost of gas contracting.

Using ONLY the solidity optimizer is not a solution to the task!

The last task requires optimizing the performance of the entire contract.

**Tasks 1-9 are worth 1 point, task 10 is worth 5 points**

## Task

Given contract `A`, we need to write a second contract named `AOptimized`.

Do not change the names of functions inside the contract!!!

Example,

```solidity
// file A.sol

contract A {
    function call() public {
        // Original function
    }
}

contract AOptimized {
    function call() public {
        // Your optimization
    }
}
```

Answers are checked manually and with the commands

```sh
forge test --match-contract ArrayLength
forge test --match-contract ArrayLength --gas-report
```

Test the same way in different contracts

Example,

```solidity
// file A.t.sol

contract ATest is Test{
    function test_call() public {
        // Original test
    }
}

contract AOptimizedTest is Test {
    function test_call() public {
        // Your test
    }
}
```

### Задание 10 — MultiSigWallet

| Функция | Оригинал | Оптимизировано | Экономия |
|---------|----------|----------------|----------|
| submitTransaction | 98 906 | 87 439 | 11 467 (11.6%) |
| confirmTransaction | 161 341 | 132 697 | 28 644 (17.8%) |
| executeTransaction | 235 002 | 179 863 | 55 139 (23.5%) |
| executionFailure | 215 540 | 180 162 | 35 378 (16.4%) |
| onlyOwnerCanSubmit | 19 918 | 12 092 | 7 826 (39.3%) |
| onlyOwnerCanConfirm | 101 389 | 86 834 | 14 555 (14.4%) |
| onlyOwnerCanExecute | 237 344 | 180 997 | 56 347 (23.7%) |

**Применённые оптимизации:**
- `mapping(address => bool)` для O(1) проверки владельца вместо O(n) перебора массива
- Упакованная структура `Transaction` (2 слота storage вместо 6): удалены избыточное поле `transactionID` и неиспользуемое `executionTimestamp`, `address + bool + uint8` упакованы в один слот
- Пользовательские ошибки вместо `require` со строками
- `unchecked` инкременты в циклах
- Кэширование чтений из storage в локальные переменные
- `private` вспомогательная функция `_executeTransaction` для устранения дублирования кода

**Дополнительные возможности оптимизации:**
- Битовая маска вместо `mapping(address => bool)` для подтверждений — хранить все подтверждения транзакции в одном `uint256`, где каждый бит соответствует владельцу. Одна операция `sstore` вместо отдельного маппинга
- Transient storage (`tstore`/`tload`, EIP-1153) — для временных данных в рамках одной транзакции (доступно с Solidity 0.8.24+)
- Assembly для горячих путей (`confirmTransaction`, `executeTransaction`) — ручное управление storage layout, прямые `sload`/`sstore`
- Minimal proxy (EIP-1167) — значительно дешевле деплой контракта через клонирование

## Общие техники дальнейшей оптимизации

| Техника | Описание | Потенциал |
|---------|----------|-----------|
| Inline assembly (Yul) | Прямой доступ к опкодам EVM, минуя проверки компилятора | 10–30% сверх текущего |
| Подбор селекторов функций | Имена функций с селекторами, начинающимися с нулевых байтов — дешевле calldata | ~60 gas на вызов |
| Transient storage (EIP-1153) | `tstore`/`tload` для временных данных — в 10 раз дешевле обычного storage | значительный для временных переменных |
| Битовые маски | Хранение нескольких bool/малых значений в одном `uint256` | экономия на `sstore`/`sload` |
| Minimal proxy (EIP-1167) | Клонирование контрактов вместо полного деплоя | экономия на деплое |

> **Примечание:** assembly-оптимизации дают ощутимый выигрыш, но значительно снижают читаемость и аудируемость кода. В production-контрактах следует находить баланс между экономией газа и безопасностью.
